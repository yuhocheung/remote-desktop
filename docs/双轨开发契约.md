# 双轨并行开发契约（Track A 媒体面 / Track B 韧性面）

> 用途：本仓库由两名开发者**并行、互相独立**推进。本文档是唯一的「对接基线」，
> 双方都只读、不修改其中标注为「冻结」的契约；各自拥有互不重叠的 crate / 子模块，
> 连同一文件都不需要改对方。

## 0. 总原则

- **只在冻结契约上对接**，其余各自为政。
- **冲突面极小**：Track A 只新增 FFI「媒体函数」（新函数名），Track B 只**在 `Message` 枚举尾部追加变体**。
  两者不碰同一份代码，无文件级 merge 冲突。
- 任意一方改动后，必须保证**另一方 CI 仍绿**（见 §6）。

```c

    int32_t button;    // 0=Left,1=Middle,2=Right,3=Back,4=Forward
```

## 5. `Message` 枚举追加位（Track B 专用，双方需同步一次下标）

当前尾部：`FileTransfer = 8`（变体共 9 个）。Track B 从**下标 9** 起继续追加，**每次追加后同步下标给 Track A**（Track A 不依赖这些变体，仅知会）：

| 下标 | 变体 | 用途 |
|---|---|---|
| 8 | `FileTransfer(FileTransferEvent)` | 文件传输（Track B 实现） |
| 9+ | （按需追加） | 其它控制面扩展 |

**铁律**：只能追加，绝不重排；新增后 `rdcore-proto` 须 `cargo test` 通过（postcard 下标一致）。

**⚠️ Dart 侧锁步（原漏写，kimi-k3 审定中发现）**：Dart 端 `flutter/lib/connection/signaling.dart` 的
`MessageType` 枚举用 `MessageType.values[idx]` 按 postcard 首字节路由。Track B 在 Rust 追加
`FileTransfer=8` 时，**必须同步在 `signaling.dart` 的 `MessageType` 枚举末尾追加 `fileTransfer`**，
否则 Dart 收到下标 8 会因 `idx >= values.length` 返回 null 而丢弃消息。该枚举由 Track B 负责追加
（协议作者即维护者），Track A 永不修改 `signaling.dart` 的 `MessageType`。

## 9. `recv_app` 使用约定（Track B 同步给 Track A，集成时必读）

**背景**：`rdcore-app` 的 `Connection::recv_app()` 对底层 DataChannel 是**单消费者**。
Track B 的 `ConnectionSupervisor`（`connection_lifecycle.rs`）在连接建立后会**独占** `recv_app()`
来监听心跳：心跳就地刷新活性，**非心跳业务消息（Input/Clipboard/FileTransfer 等）经内部
`broadcast` 通道一式多投**（`recv_business` 与 `Connection::recv_input` 各持一个订阅者）。

**约定**：一旦某条连接挂上了 `ConnectionSupervisor`——

- ✅ 上层（含 Track A 的输入/剪贴板处理）**必须**经 `ConnectionSupervisor::recv_business()` 取业务消息。
- ❌ 上层**不得**再直接调 `conn.recv_app()`——否则会与 supervisor 争抢同一条通道
  （心跳被吃掉导致误判 Dead，或业务消息被心跳环吞掉丢失）。

**不受影响**：未挂 supervisor 的连接（如 Track A 的媒体通路、`send_app`/`recv_media`）不受此约束；
本约定只约束"已启用 `ConnectionSupervisor` 的连接"的**控制消息接收**路径。

> 该约定已在 Track B 的 `e2e_lifecycle.rs` 经真实 WebRTC 链路验证
> （`supervisor_keeps_up_with_heartbeats_and_filters_business`：业务消息正确转发、心跳被过滤）。

**P0 修复（WorkBuddy 已落地，2026-07-20）**：为彻底消除"同一条连接上 supervisor 与 Track A 争抢 `recv_app`"的潜伏冲突，`Connection` 现提供：
- `Connection::set_business_receiver(rx: broadcast::Receiver<AppMessage>)`（async）：由 supervisor 注入业务转发通道。
- `Connection::recv_input` 改造为：**注入后优先从该 `broadcast` 通道读 `Input`，不再调 `recv_app`**；未注入时回退到 `recv_app`（孤立场景 / 单元测试）。
- 回归测试 `e2e_media_input::recv_input_routes_via_injected_business_channel` 已验证该路由。
- 选用 `broadcast` 而非 `mpsc`：业务消息需被 supervisor（`recv_business`）与 `Connection`（`recv_input`）**两个消费者**各自独立接收，broadcast 多订阅者语义正好满足。

**B 侧注入（Track B 在 `ConnectionSupervisor::start` 内，已于 2026-07-20 落地闭环）**：
```rust
// 业务消息走 broadcast，supervisor 与 Connection::recv_input 各持一个订阅者。
let (biz_tx, biz_rx) = tokio::sync::broadcast::channel::<AppMessage>(64);
// …原有 recv_app 环把业务消息改发 biz_tx，recv_business 改为 drain biz_rx…
conn.set_business_receiver(biz_tx.subscribe()).await; // ← 注入给 Connection::recv_input
```
如此：supervisor 独占 `recv_app`（心跳就地处理），`Connection::recv_input` 经 broadcast 业务通道收 Input，两侧物理零冲突（A 改 `Connection`，B 改 `supervisor::start`，互不 `use` 对方符号）。

> **闭环现状（2026-07-20，WorkBuddy 复审确认）**：A 侧 `Connection::set_business_receiver` + `recv_input` 走 broadcast，与 B 侧 `ConnectionSupervisor::start` 的 `broadcast::channel(64)` + `conn.set_business_receiver(biz_tx.subscribe())` 均已落地；两订阅者实为 `recv_business`（`biz_rx`）与 `Connection::recv_input`（注入的 `biz_tx.subscribe()`）。`cargo build --workspace` 通过、`cargo test -p rdcore-app` 全绿（A 媒体/输入 + B lifecycle 集成），同连接挂 supervisor 不再争抢 `recv_app`，§9 潜伏冲突彻底消除。

**✅ B 侧已闭环（kimi-k3，2026-07-20）**：上述那一行已落地——`ConnectionSupervisor::start`
改为 `async`，内部建 `broadcast::channel(64)`、recv_app 环把业务消息 `biz_tx.send`、并
`conn.set_business_receiver(biz_tx.subscribe()).await` 注入；`recv_business` 改为 broadcast 语义
（`Lagged` 跳过滞留、`Closed` 返回 `None`）。验证：`e2e_lifecycle`（2/2，含 supervisor 心跳/
判 Dead）与 `e2e_media_input::recv_input_routes_via_injected_business_channel` 全绿。
