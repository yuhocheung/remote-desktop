- **佐证方向正确**：`flutter/NATIVE_FFI.md` 文档写的预期路径正是 `ios/rdcore/librdcore_ffi.a`；原脚本是唯一离群项。

### 2.3 ⚠️ 仍需 B 知晓的 iOS 风险（非 A 单方面能全验）
1. **交叉编译环境**：本沙箱无 macOS/iOS SDK，无法实跑 `cargo rustc --target aarch64-apple-ios`。但 `target/aarch64-apple-ios/debug/librdcore_ffi.a` 已存在 → FFI 的 iOS staticlib 至少成功编过一次。
2. **FFI 依赖边界**：`rdcore-ffi` **不依赖 `rdcore-rtc`(webrtc-rs)**，最重的 iOS 不兼容依赖未进入 FFI 树 → 风险低于预期。但它拉了 `rdcore-capture`，**iOS 必须 `default`(Null 后端) 构建，绝不能开 `real`(scrap/enigo)**，否则编译爆。
3. **openh264 C 核心**：encode/decode 依赖 `openh264` 的 C 核心，交叉编译需 iOS clang/SDK；全新非 Mac CI 可能卡此处。
4. **双脚本漂移**：存在两份 `build_ffi.sh`——`tool/build_ffi.sh`(Track A 维护，按 header 注释 B6 应调用本脚本) 与 `flutter/tool/build_ffi.sh`(Track B 副本，已输出正确路径，但多编了 Apple 已废弃的 `x86_64-apple-ios` 三切片)。**建议**：以 `tool/build_ffi.sh` 为准，删除/别名化 `flutter/tool/build_ffi.sh`，避免后续漂移。

## 3. 🆕 A5（Windows Host Agent 打包）真实状态——**✅ 已完成（纯 Rust 服务）**

计划验收是"Windows 启动 Agent → 注册信令 → iPhone 配对后看到画面并控制"。现状：

- ✅ **Flutter Windows Viewer** 侧 FFI dll 打包（`flutter/windows/CMakeLists.txt` 引用 `rdcore/rdcore_ffi.dll`，dll 已放置）。
- ✅ **被控端 Host Agent 整套已实现**（`core/crates/rdcore-desktop`，纯 Rust 服务）：
  - 子命令 `run`/`install`/`uninstall`/`service`（clap 解析）；`service` feature 经 `windows-service` 0.6 注册 SCM、处理 Stop/Shutdown。
  - 编排（`agent.rs`）：`create_local_identity` + `create_pairing()` → 写同机 `SIGNALING_TOKEN_DB` token 文件（**B2 对接点**，signaling-svc 每次握手前 `reload_from_file` 刷新内存 `TokenStore`）→ `Connection::new_host` → `establish(shutdown, Grant)`（**Host 侧阻塞等 Viewer Offer**）→ `start_capture(|| ScrapCaptureSource::new(), fps)`（**factory 形式**：专用 OS 线程就地构造 `!Send` 捕获源，编码帧经 `mpsc` 交给 tokio 发送端）→ `EnigoInputInjector` 输入注入（headless 用 `NullInputInjector`）→ `rdcore-banner` 横幅推送（独立进程 + UDP 回环 48173，Host 仅作推送方）。
  - 验证：默认 + `service` feature 均 `cargo check`/`test`/`clippy -D warnings` 全绿，`rdcore-desktop` 7 单测通过；`cargo build --workspace` 无回归（`start_capture` factory 化对 `rdcore-app` 测试无影响）。

### 3.1 A5 与 Track B 的接口耦合（已在 A5 落地中敲定）
Agent 形态涉及共享组件，须双方共识，否则 B 先写配对 UI 而 A 的 Agent 接口对不上：
- **信令客户端**：Host Agent 用哪份 signaling client？是复用 A 的 `rdcore-signaling::SignalingClient`，还是 B 在 `rdcore-app` 编排层统一提供？（A 侧 `rdcore-app::Connection` 已放宽签名，可承载。）
- **配对流程**：A5 的"注册/上线/等待配对"需对齐 B 的配对状态机（G 文件/剪贴板也复用同一状态机）。
- **横幅**：Host Agent 须拉起 `rdcore-banner` 独立进程（OS 不可伪造），A 已实现，B 的 UI 不应重复造。
- **FFI 暴露面**：Host Agent 若以 Rust 原生二进制（非 Flutter）运行，则控制面逻辑走 `rdcore-app::Connection`；若要经 Flutter 控制，则 B 需在 Dart 侧编排。这决定了 A5 是"纯 Rust 服务"还是"Rust 核心 + Flutter 壳"。

**已闭环**：形态=纯 Rust 服务；信令入口=复用 A 的 `rdcore-signaling::SignalingClient` + `rdcore-app::Connection`（A0 已放宽）；配对入口=`Connection::create_pairing()` 拿 token 推 B2 per-session token 库；横幅=复用 `rdcore-banner` 独立进程。A5 已开工并完成，详见 roadmap.md（A5 100%）。

### 3.2 🆕 受控端形态收敛（2026-07-24 决议，当日二次修订）

Flutter 桌面/移动端 UI 上曾暴露「作为被控端」入口（`home_screen.dart` 两处按钮 + `pairing_page.dart` Host 模式，走 FFI `rdcore_connection_new_host`）。与 `rdcore-desktop` Host Agent 在产品形态上重叠，决议：

- **所有平台的正式受控端 = `rdcore-desktop`**（SCM 服务生命周期 + 不可伪造横幅 + headless 能力），**Flutter 端在全部平台（Windows / macOS / Linux / Web / 移动端）均为纯 Viewer（控制端）**。
- **实现**（二次修订后）：`flutter/lib/ui/home_screen.dart` 直接移除两处「作为被控端」按钮与 `_startHostPairing` 导航方法——Host 模式配对页由此不再有 UI 入口，release 构建随 tree-shaking 一并裁剪。初版的 `_showHostEntry` 平台门控（仅 Windows/Web 隐藏）已被本决议取代。
- **保留**：`ConnectionManager.startHost()` / FFI Host 面（同意门控、抓屏、输入注入）原样保留，供同机回环联调与未来非 Windows 受控形态复用；底层 Host 引擎仍是共享的 `rdcore-app` / `rdcore-session`，无双份实现。
- **macOS 受控端**：Flutter 壳的 FFI 抓屏仅 `NullCaptureSource` 合成帧（真实抓屏在 `rdcore-desktop` 的 `real` 后端，不经 FFI），且 `rdcore-desktop` 服务封装为 Windows 导向（横幅/SCM 大量 `cfg(windows)`）；macOS 受控端需在 Mac 上先做 `rdcore-desktop run` 适配验证（scrap/enigo 后端 + 屏幕录制/辅助功能授权 + launchd）。
- **验证**：`flutter analyze` 无告警；`scripts/build_flutter_win.cmd` 全量构建通过，exe 实跑确认入口消失（二次修订后已重新构建验证）。
